1

SQSメッセージ送信

システム A が queue-from-a へ JSON を送信。message_id を必ず含める。

2

Lambda自動起動

SQS の Event Source Mapping が Lambda をトリガー。最大 BatchSize=10 件をまとめて処理。

3

DynamoDBべき等性チェック

message_id をキーに DynamoDB を確認。SQS は at-least-once 配信のため、同じメッセージが2回来ても1回だけ処理する。

4

S3受信アーカイブ

変換前の生データを S3 に保存（a_to_b/YYYY/MM/DD/{id}_received.json）。監査・デバッグ用。

5

Lambdaデータ変換

transformer.py でフィールドマッピング。例: item_code→product_id、qty×price→amount

6

SQS変換済みデータを送信

queue-to-b へ送信。システム B は自分のペースでこのキューから受信する。

7

DynamoDB完了記録

ステータスを done に更新。TTL=24時間で自動削除。

batchItemFailures とは？
Lambda がバッチ内の失敗レコードだけを報告する機能。
10件中1件が失敗しても、残り9件は正常に削除される。
失敗した1件だけがリトライ・DLQ の対象になる。
DLQ にメッセージが溜まったら？
SQS コンソール → dlq-from-a を確認。
「メッセージをポーリング」でエラー内容を確認し、
修正後に元のキューへ手動再送する。

なぜインターネットを経由しないのか？
Lambda は Private Subnet に配置され、Internet Gateway も NAT Gateway もありません。
SQS / DynamoDB / S3 へのアクセスはすべて VPC Endpoint 経由でプライベートに通信します。
システム A・B との通信も VPC Peering または Transit Gateway 経由で、AWS バックボーンネットワーク内で完結します。
